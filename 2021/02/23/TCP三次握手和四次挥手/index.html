TCP三次握手和四次挥手?

创建时间2021-02-23更新时间2021-02-25字数统计1.6k阅读时间8m

TCP标识

  • SYN(建立联机)
  • ACK(确认)
  • PSH(传送)
  • FIN(结束)
  • RST(重置)
  • URG(紧急)

TCP三次握手？

三次握手的过程（a是客户机，b是服务端）

  • 第一次(A—>B)，SYN=1，seq=x

第一次客户端发送syn = 1 就表示要建立连接，并且随机产生了一个序列号seq作为数据包的标识发送给对方。

⚠️注意：只有当接收方接收到syn为1的syn包时，才可建立连接，别无他法。

  • 第二次(B—>A)，SYN=1，ACK=1，seq=y，ack=x+1

第二次服务端发送SYN/ACK都设为1的SYN/ACK包，表明对第一个SYN包的确认，并且继续握手连接。同样的，随机生成自己数据包的序列号seq，并且通过ack对A的数据包进行确认，表明确确实实的收到了A发送过来的序列号为x的包，期望下次收到序列为x+1。

⚠️注意：这个ack有确认和期望序列号的作用，确认收到A的数据，期望A下次的序列号时x+1

  • 第三次(A—>B)，ACK=1，seq=x+1,ack=y+1

第三次客户端发送一个ACK=1的ACK包，通知A连接已建立。同时发送序列号seq以及确认号ack，这个确认号在这里的意义是表明确确实实的收到了上一次服务端发送过来的数据包。服务端收到该数据包后就明白上次发给客户端的数据对方已经收到，连接建立完成，等待客户端发送数据。

为什么需要三次握手？

tcp三次握手最主要的原因是发送方确认发送的数据在接收方已经正确的接受到，两者之间通过序列号和确认号来通知。

假如是两次握手的话，仅仅连接发起方选择的序列号能够得到确认，而另一方选择的序列号得不到确认。

因此三次握手实际上是确认两端收发功能都正常的必要手段。

另外，网络的资源是宝贵的，如果只有两次握手，当存在失效的连接请求到达服务端时，会产生错误。假设客户端发送连接请求到服务端，但因种种原因使得请求报文滞留在网络中没办法顺利发送过去。而在这之后，失效的请求发送到服务器上，服务器就认为客户端想建立连接，发送确认报文，此时两次握手就建立起了连接，服务端等待客户端发送数据。
但请求是失效的请求，客户端是不会理睬服务端的确认信息，也不会发送任何数据报文给服务端。而服务端却在傻傻的等着，此时很多宝贵的资源就白白浪费掉了。
如果三次握手的话，服务端没收到客户端的确认报文就没办法建立起连接。

TCP四次挥手？

  1. c发送断开连接请求报文,FIN设为1的FIN报文，停止发送数据，并且携带一个序列号u（该序列号为前面传过来的数据的最后一个字节的序号+1）
  2. s发送一个ACK确认报文，表示我收到了你的断开连接请求，并且携带确认号ack为u+1，以及自己的序列号v。此时服务端处于半关闭的状态（close-wait），tcp服务器通知上层应用层，客户端打算释放连接了，即客户端没有数据要发送了，但此时服务端可能仍有数据要发送，并且客户端依然需要接受。
  3. 若服务端有数据要发送并且发送完数据后，则同意断开连接，发送FIN为1的FIN报文，携带序列号为w，确认号为u+1，此时服务端就进入了最后确认状态，等待客户端确认
  4. 客户端收到FIN终止连接报文段后，发送ACK报文段，并且携带序列号为u+1，确认号为w+1。此时发送完确认报文段后，客户端并不立即关闭连接而是等待2MSL时间后才进入closed状态。
    服务端只要收到了客户端的确认，则立即进入关闭连接 的状态。

为什么 客户端 要进入 TIME-WAIT 状态，等待 2MSL 时间后才进入 CLOSED 状态？
为了保证 服务端 能收到 客户端 的确认应答。若 客户端 发完确认应答后直接进入 CLOSED 状态，如果确认应答因为网络问题一直没有到达，那么会造成 服务端 不能正常关闭。

在此期间，如果ACK报文丢失了，客户端会重新发送。

为什么需要四次挥手？

为了确保数据能够完成传输。

关闭连接时，当收到对方的FIN报文通知时，它仅仅表示对方没有数据发送给你了；但未必你所有的数据都全部发送给对方了，所以你可以未必会马上会关闭SOCKET,也即你可能还需要发送一些数据给对方之后，再发送FIN报文给对方来表示你同意现在可以关闭连接了，所以它这里的ACK报文和FIN报文多数情况下都是分开发送的。

tcp我握手的时候为何ACK(确认)和SYN(建立连接)是一起发送。挥手的时候为什么是分开的时候发送呢？

因为当 Server 端收到 Client 端的 SYN 连接请求报文后，可以直接发送 SYN+ACK 报文。其中 ACK 报文是用来应答的，SYN 报文是用来同步的。但是关闭连接时，当 Server 端收到 FIN 报文时，很可能并不会立即关闭 SOCKET，所以只能先回复一个 ACK 报文，告诉 Client 端，”你发的FIN报文我收到了”。只有等到我 Server 端所有的报文都发送完了，我才能发送 FIN 报文，因此不能一起发送。故需要四步握手。
