何为模块化

模块化是指把一个复杂的系统分解成一个一个的模块。
块的内部数据与实现是私有的，通过向外暴露一些接口（方法）与其他模块通信。

自从前后端分离后，前端就变成了一种工程化的项目，因此模块化就显得十分有必要：一个工程是由许多个模块组成的，
只有将工程中的各个部分抽离成不同的模块，才能更好的进行工作。

因此，模块化开发具有以下的优点：

  • 代码复用，让我们更方便地进行代码管理、同时也便于后面代码的修改和维护。
  • 一个单独的文件就是一个模块，是一个单独的作用域，只向外暴露特定的变量和函数。这样可以避免污染全局变量，减少变量命名冲突。js模块化规范有：CommonJS、AMD、CMD、ES6的模块系统，本篇文章我们着重讲解ES6的模块化。

在此之前我们先讲一下在没有这些模块化规范之前，是如何在代码里面使用模块化的，这里主要讲解**IIFE模式：匿名函数自调用(闭包)**。

它是通过一个匿名函数将数据私有化，将操作数据的行为封装到方法里面，并且通过暴露接口到window对象上来实现。

并且引入多个文件，造成了多次请求，浪费资源。

几种模块化规范

CommonJS

是服务器端模块的规范，由nodejs推广使用。该规范的核心思想是：允许模块通过require方法来同步加载所要依赖的其他模块，然后通过 exports 或 module.exports 来导出需要暴露的接口。

Node 应用由模块组成，每个文件就是一个模块，有自己的作用域。
在一个文件里面定义的变量、函数、类，都是私有的，对其他文件不可见。
在服务器端，模块的加载是运行时同步加载的；在浏览器端，模块需要提前编译打包处理。

⚠️模块可以多次加载，但是只会在第一次加载时运行一次，然后运行结果就被缓存了，以后再加载，就直接读取缓存结果。要想让模块再次运行，必须清除缓存。

我们在服务端通过 node b.js 就可以运行代码了。

需要注意的是，如果是运行在浏览器端，还需要将文件先打包编译，然后放到页面中的script标签中去才能运行起来。
我们这里使用Browserify工具进行编译打包，它可以让代码在浏览器环境下像node.js一样遵循commonJS规范，所以我们就可以在浏览器下愉快的玩耍commonJS模块化了！

  1. 安装Browserify进行编译打包：npm i Browserify -D
  2. 在package.json中写build脚本: "build": "browserify b.js -o ./dist/bundle.js"（如果是全局安装browserify的话直接在终端运行即可）
  3. npm run build
  4. 将bundle.js插入到index.html页面就可以运行起来了

AMD

AMD规范是非同步加载模块，允许指定回调函数。

CommonJS规范加载模块是同步的，也就是说，只有加载完成，才能执行后面的操作。
由于Node.js主要用于服务器编程，模块文件一般都已经存在于本地硬盘，所以加载起来比较快，不用考虑非同步加载的方式，所以CommonJS规范比较适用。但是，如果是浏览器环境，要从服务器端加载模块，这时就必须采用非同步模式，因此浏览器端一般采用AMD规范。此外AMD规范比CommonJS规范在浏览器端实现要来着早。

这个就是AMD规范，通过define来声明模块，并且在其中可以注入其他声明过的模块，通过require来引入模块并使用他们。

AMD规范在浏览器端的实现就是Require.js，具体使用方法可以自行去查找。

CMD规范

CMD规范专门用于浏览器端，模块的加载是异步的，模块使用时才会加载执行。CMD规范整合了CommonJS和AMD规范的特点。

ES6模块化

ES6在语言的层面上实现了模块化。浏览器厂商和 Node.js 都宣布要原生支持该规范。它将逐渐取代 CommonJS 和 AMD 规范，成为浏览器和服务器通用的模块解决方案。

ES6 模块的设计思想是尽量的静态化，使得编译时就能确定模块的依赖关系，以及输入和输出的变量。CommonJS 和 AMD 模块，都只能在运行时确定这些东西。比如，CommonJS 模块就是对象，输入时必须查找对象属性。

语法：export命令用于规定模块的对外接口，import命令用于输入其他模块提供的功能。

注意导出的时候有两种形式：

  1. 具名导出。则在引入的时候需要知道导出的变量或者函数名，并且可以按需引入。
  2. 导出整个对象。通过default关键字来导出整个对象，然后可以通过这个对象来进行操作，该对象可以指定任意名字作为默认输出的变量。

由于模块化是ES6的语法，浏览器不支持，我们需要将它转换成浏览器认识的ES5的代码，并且通过编译打包输出文件用于插入页面运行。

此处使用babel进行转码，使用Browserify（也可以使用webpack）进行编译打包,babel的具体文档可以去官方查阅，这里我们直接进入使用。

⚠️CommonJS与ES6的差异

  • CommonJS 模块输出的是一个值的拷贝，ES6 模块输出的是值的引用。
  • CommonJS 模块是运行时加载，ES6 模块是编译时输出接口。

而在ES6中，它是一种值的引用，因此原模块的count变量一改，这边就能同步改变。

接着对第二个差异作解释：

因为 CommonJS 加载的是一个对象（即module.exports属性），该对象只有在脚本运行完才会生成。而 ES6 模块不是对象，它的对外接口只是一种静态定义，在代码静态解析阶段就会生成。

⚠️ES6模块化需要注意的几点

在script标签中默认是不能使用模块化写法的，在script标签中声明type=”module”即可以开启模块写法。

模块化的延迟解析和严格模式:
延迟解析： 带有模块化类型的script标签总是会在最后加载，它会等到所有模块加载完才加载自己的。同defer效果类似。
严格模式： 使用模块化会自动将此script的脚本定义在严格模式下，因此会遵循严格模式的规则。

模块化的预解析:
在引入模块的时候，模块的初始化只会做一次，因此后面引入的时候不会再进行初始化。
譬如我们有一个moudle.js:console.log('ivan');,然后在其他文件中引入它:import {} from './moudle.js';import {} from './moudle.js';import {} from './moudle.js'。
它们只会打印出一次结果：它原先引入的时候预解析了一次，之后再引入就不会再执行了。
我们很容易想到，在一个模块中，我们从后台抓取了数据。第一次引入的时候数据就抓取了，如果第二次引入还要抓取，那就十分没必要了，这样是有好处的。

模块的批量导入:
假如我们想导入整个模块，那么可以用 * as + 别名 : import * as api from ‘./moudle.js’。调用的时候加上前缀就行了，但通常情况下还是具名导入比较好。

模块化别名:
在引入或者导出的时候可以通过as来给导入（导出）的变量或对象起别名，这样就可以避免一些模块中的重名变量。

默认导出和具名导出:
一个模块中不可以导出两个default！在含有default的模块中，也可以实现原来的按需导出和导入（具名）。

按需动态加载模块:
import from 导入模块只能放在最外层，不能放在块中，否则会报错。但是有时候我们要实现一种动态加载的功能，譬如当我点击某个按钮的时候，才会去加载查询课程的模块。这个时候这种写法放在事件监听函数里就会报错。
所以我们可以用 import( ) 这种方式， 这个函数会加载对应路径的模块，并且返回一个promise对象。且该对象的状态时fufilled的，因此直接使用then方法可以运行回调，回调接受的参数是模块对象。

总结

  • CommonJS规范主要用于服务端编程，加载模块是同步的，这并不适合在浏览器环境，因为同步意味着阻塞加载，浏览器资源是异步加载的，因此有了AMD CMD解决方案。
  • AMD规范在浏览器环境中异步加载模块，而且可以并行加载多个模块。不过，AMD规范开发成本高，代码的阅读和书写比较困难，模块定义方式的语义不顺畅。
  • CMD规范与AMD规范很相似，都用于浏览器编程，依赖就近，延迟执行，可以很容易在Node.js中运行。不过，依赖SPM 打包，模块的加载逻辑偏重。
  • ES6 在语言标准的层面上，实现了模块功能，而且实现得相当简单，完全可以取代 CommonJS 和 AMD 规范，成为浏览器和服务器通用的模块解决方案。
