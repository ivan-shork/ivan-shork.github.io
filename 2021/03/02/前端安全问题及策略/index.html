XSS跨站脚本攻击

XSS攻击全称跨站脚本攻击，XSS是一种在web应用中的计算机安全漏洞，它允许恶意web用户将代码植入到提供给其它用户使用的页面中。

常见的 XSS 攻击有三种：

  • DOM-based型
  • 反射型
  • 持久型

反射型、DOM-based 型可以归类为非持久型 XSS 攻击。存储型归类为持久型 XSS 攻击。

DOM-based型

基于 DOM 的 XSS 攻击是指通过恶意脚本修改页面的 DOM 结构，是纯粹发生在客户端的攻击，是javascript编写存在漏洞造成的。
它不需要服务器解析响应的直接参与，触发XSS靠的是浏览器端的DOM解析。

主要有：

  • DOM的操作：document.create document.body …
  • 输出html内容: document.write document.writeln innerHTML outerHTML …
  • document相关链接: document.url document.location(直接给href、host、hostname属性赋值)…
  • 窗口:document.open window.open window.location.href=…
  • 脚本执行: eval setTimeout setInterval

这些文档或者全局的api，都能把字符串作为代码运行。如果不可信的数据拼接到字符串中传递给这些 API，很容易产生安全隐患。

反射型

反射型的 XSS 攻击，主要是由于服务端接收到客户端的不安全输入，在客户端触发执行从而发起 Web 攻击。

具体而言，反射型 XSS 只是简单地把用户输入的数据 “反射” 给浏览器，这种攻击方式往往需要攻击者诱使用户点击一个恶意链接，或者提交一个表单，或者进入一个恶意网站时，注入脚本进入被攻击者的网站。这是一种非持久型的攻击。

比如在提交表单的文本框中输入<script>alert('javascript')</script>，这样服务端收到这个提交的文本后没有进行过滤就返回给客户端解析，会引发安全漏洞。

持久性

基于存储的 XSS 攻击，是通过提交带有恶意脚本的内容存储在服务器上，在下一次拉取内容交互的时候会产生安全攻击。一般提交的内容都是通过一些富文本编辑器编辑的，很容易插入危险代码。

譬如有恶意者在编写博客的时候注入了一段恶意的代码，并将这篇文章post到服务端去。服务端将这篇带有恶意代码的文章存储在数据库中，这样一来，当其他人点击访问这篇文章的时候，就会受到该恶意代码的攻击。这只是一个简单例子。

防御XSS

XSS攻击的主要特点就是:

  • 攻击者注入恶意代码
  • 浏览器执行恶意代码

Web安全头

  • CSP：W3C 的 Content Security Policy，简称 CSP，主要是用来定义页面可以加载哪些资源，减少 XSS 的发生。要配置 CSP , 需要对 CSP 的 policy 策略有了解，具体细节可以参考 CSP 是什么。
  • X-Download-Options: noopen：默认开启，禁用 IE 下下载框 Open 按钮，防止 IE 下下载文件默认被打开 XSS。
  • X-Content-Type-Options: nosniff：禁用 IE8 自动嗅探 mime 功能例如 text/plain 却当成 text/html 渲染，特别当本站点 server 的内容未必可信的时候。
  • X-XSS-Protection：IE 提供的一些 XSS 检测与防范，默认开启

这里着重说一下CSP:
内容安全策略(CSP)是一个额外的安全层，用于检测并削弱某些特定类型的攻击，包括跨站脚本 (XSS) 和数据注入攻击等。无论是数据盗取、网站内容污染还是散发恶意软件，这些攻击都是主要的手段。

说白了，就是对站点的资源加载定义不同的规则，允许哪些类型的资源可以从哪些源下进行加载，应用它的站点会仅仅执行从白名单域获取到的脚本文件。

开启CSP:

  1. 服务器设置 HTTP Header 中的Content-Security-Policy
  2. 设置meta标签的方式

这里列举一下简单的例子

  • 只允许加载本站资源:Content-Security-Policy: default-src ‘self’
  • 只允许加载 HTTPS 协议图片:Content-Security-Policy: img-src https://*
  • 允许加载任何来源框架:Content-Security-Policy: child-src 'none'

Cookie 的 HTTPOnly

将重要的Cookie标记为HTTPOnly,浏览器将禁止页面的 Javascript 访问带有 HttpOnly 属性的 Cookie，因此不能通过客户端脚本读取和修改。

攻击者可以通过注入恶意脚本获取用户的 Cookie 信息。通常 Cookie 中都包含了用户的登录凭证信息，攻击者在获取到 Cookie 之后，则可以发起 Cookie 劫持攻击。

严格来说，HttpOnly 并非阻止 XSS 攻击，而是能阻止 XSS 攻击后的 Cookie 劫持攻击。

CSRF攻击

CSRF的全称是Cross Site Request Forgery，即跨域请求伪造。

主要在于用户访问受信任的站点并保存cookie在本地，此时用户处于登陆状态，在登陆状态下用户又去访问了一个恶意网站，恶意网站通过窃取的cookie冒充用户，从而达到能够请求站点的目的。

按照请求类型可分为GET类型和PSOT类型的CSRF攻击。

若按照攻击类型分类，可分享HTML CSRF攻击、JSON HiJacking 攻击和Flash CSRF攻击。

HTML CSRF攻击

这一类是最普遍的CSRF攻击，HTML中能设置src/href等链接地址的标签都可以发起一个GET请求，如a、img、iframe、script、video、audio等…

还有通过JavaScript动态生成的标签对象或CSS对象发起的GET请求，而发出POST请求只能通过form提交。

JSON HiJacking 攻击

JSON HiJacking 非常经典，攻击过程是CSRF，不过是对AJAX 响应中 最长久的JSON数据进行的挟持攻击。

一些应用中有的接口是使用JSONP来返回json数据的，如果在这个接口中没有做相应的防护，那么攻击者冒充用户请求从而获得敏感数据，是一件十分容易的事情。

JSONP就是在前端定义一个函数并作为参数包含在script脚本的src中请求后端，后端处理完后将json数据包在函数参数里面并返回给前端。

前端接收到后执行相应的函数获取数据。

因此JSON HiJacking的流程大概是这样：

  1. 攻击者在虚假页面中加入script标签请求服务器，并且重写了回调函数。
  2. 用户访问恶意网站，恶意网站冒充用户向服务器发起了请求。
  3. 服务器返回了用户的敏感数据。
  4. 恶意网站执行自己重写的回调函数，在里面执行自己的恶意代码（一般是将用户的敏感数据发送到自己的服务器上）。

防御CSRF

头部referer

http头中有一个字段叫’Referer’，它记录了该 HTTP 请求的来源地址。
譬如有这样一个网站”http://example.com/",服务器可以通过验证请求头部携带的Referer字段是不是以'example.com'域名开头的。如果是，则证明请求是合法的。如果不是，则拒绝该请求。

token

token机制是前端重要的一个身份验证机制，通过给每个用户颁发一个”令牌”来确定用户的身份。

我们可以在请求头部自定义一个存放token信息的字段，该token由服务器下发给客户端，客户端每次请求时都携带该请求头。如果该请求头对应上了则放行请求，否则拒绝该请求。
